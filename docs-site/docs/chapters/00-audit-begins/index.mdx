## The Visual Tour

Launch CheckBank and take a moment to look at it. Really look.

The login screen greets you with a clean teal bank icon, a bold "CheckBank" title, and a welcoming subtitle. Two input fields sit neatly below — email and password — followed by a wide teal "Go" button. A "Forgot password?" link sits underneath in understated grey text. The whole screen is centred, well-spaced, breathing room everywhere. It feels considered.

Sign in and the dashboard unfolds: account summary cards showing balances, a recent transactions list with category icons, amounts formatted in neat columns. Navigate to the Transactions tab and you get date-based filters, a search bar, and a scrollable list with alternating visual weights. The Transfer screen walks you through a stepper — choose recipient, enter amount, confirm. Settings gives you toggles for notifications and biometric login, a theme selector, and sign-out.

The colour palette is consistent. The typography is hierarchical. The spacing is deliberate. If you showed this to a product review meeting, people would nod approvingly.

If you stopped here, you would ship it.

## The Audit Checklist

Every accessibility failure in CheckBank maps to a category. Each category gets its own chapter. Each chapter follows the same flow: write a failing test, watch it go red, fix the code, watch it go green.

Here is the full audit:

| # | Failure Category | What's Wrong | Chapter |
|---|---|---|---|
| 1 | Missing labels | Interactive elements without semantic labels | [Ch 1](/chapters/labels) |
| 2 | Missing roles & states | Buttons without roles, toggles without state announcements | [Ch 2](/chapters/roles-states) |
| 3 | Contrast failures | Text colours below WCAG AA thresholds | [Ch 3](/chapters/contrast) |
| 4 | Undersized targets | Touch targets below 48 x 48 dp | [Ch 4](/chapters/sizing) |
| 5 | Focus chaos | No logical tab order, focus does not follow action | [Ch 5](/chapters/focus) |
| 6 | Silent updates | Dynamic content that changes without announcing | [Ch 6](/chapters/announcements) |
| 7 | Journey gaps | Cross-screen flows that break for AT users | [Ch 7](/chapters/integration) |

Your mission: write an automated test for every row in this table. By the final chapter, CheckBank will have a complete accessibility test suite that runs on every pull request.

## Setting Up the Test Infrastructure

Before we write our first real accessibility test in Chapter 1, we need the plumbing in place. This section sets up the directory structure and verifies that the Flutter test framework can enable the semantics tree.

<div className="cb-step">
  <div className="cb-step__badge">1</div>
  <div className="cb-step__content">
    <div className="cb-step__title">Create the test directory</div>

All accessibility tests will live in their own directory, separate from any unit or widget tests you might write later.

```bash title="Terminal"
mkdir -p test/accessibility
```

  </div>
</div>

<div className="cb-step">
  <div className="cb-step__badge">2</div>
  <div className="cb-step__content">
    <div className="cb-step__title">Understand `tester.ensureSemantics()`</div>

Flutter widget tests do not enable the semantics tree by default. Without semantics, all the accessibility matchers and finders are blind — they cannot query labels, roles, or states because that data does not exist in the test environment.

Calling `tester.ensureSemantics()` flips the switch. It tells the framework: "Build the full semantics tree for this test, exactly as a real screen reader would see it." The method returns a `SemanticsHandle` — a disposable resource you must clean up when the test ends.

Every accessibility test you write in this tutorial will start with this line:

```dart
final handle = tester.ensureSemantics();
```

And end with:

```dart
handle.dispose();
```

Think of it as opening and closing a connection to the accessibility layer.

  </div>
</div>

<div className="cb-step">
  <div className="cb-step__badge">3</div>
  <div className="cb-step__content">
    <div className="cb-step__title">Write the smoke test</div>

Create the following file. This test does not check any accessibility properties yet — it simply verifies the login screen renders and the semantics tree is available.

<div className="cb-callout cb-callout--blue">
  <div className="cb-callout__header">Write This Test</div>
  <p>Create <code>test/accessibility/audit_smoke_test.dart</code> with the code below.</p>
</div>

```dart title="test/accessibility/audit_smoke_test.dart"
import 'package:flutter/material.dart';
import 'package:flutter_test/flutter_test.dart';
import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'package:checkbank/screens/login_screen.dart';

void main() {
  group('Audit smoke test', () {
    testWidgets('login screen renders with semantics enabled', (tester) async {
      final handle = tester.ensureSemantics();

      await tester.pumpWidget(
        const ProviderScope(
          child: MaterialApp(home: LoginScreen()),
        ),
      );

      // The app renders — but are the semantics correct?
      // We'll find out in the next chapters.
      expect(find.text('CheckBank'), findsOneWidget);
      expect(find.text('Go'), findsOneWidget);

      handle.dispose();
    });
  });
}
```

Let us walk through what this test does:

1. **`tester.ensureSemantics()`** — enables the semantics tree and gives us a handle to dispose later.
2. **`ProviderScope`** — CheckBank uses Riverpod for state management. Every screen needs a `ProviderScope` ancestor.
3. **`MaterialApp(home: LoginScreen())`** — wraps the screen in a minimal Material app so themes and navigation work.
4. **`find.text('CheckBank')`** — verifies the branding text rendered.
5. **`find.text('Go')`** — verifies the submit button rendered.
6. **`handle.dispose()`** — cleans up the semantics tree.

  </div>
</div>

But hold on. Read the test again carefully. What does it actually prove about accessibility?

Nothing.

It checks that the text "CheckBank" and "Go" appear on screen — the same things your eyes can see. It says nothing about whether a screen reader can identify the email field, whether the button has a meaningful label, or whether errors get announced.

This test is green, but it tells us nothing about accessibility. That changes in Chapter 1.

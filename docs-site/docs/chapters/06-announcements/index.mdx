# Chapter 6: Live Regions and Announcements

> *"If content changes on screen and the screen reader says nothing, did it really change?"*

**Estimated time:** ~30 minutes | **Focus:** Login + Notifications | **Branch:** `chapter-6-announcements`

---

## What Is a Live Region?

A live region is a section of the UI that screen readers **monitor for changes**. When the content inside a live region updates, the screen reader automatically announces the new content -- even if the user's focus is elsewhere on the screen.

Without a live region, dynamic changes are invisible to screen reader users. An error message might flash on screen, a notification count might update, a progress indicator might complete -- but the screen reader stays silent. The user has no idea anything happened.

WCAG 4.1.3 (Status Messages) requires that status messages can be programmatically determined through role or properties so that they are presented to the user by assistive technologies without receiving focus.

### When to Use Live Regions

| Scenario | Live Region? | Reason |
|----------|-------------|--------|
| Error message appears after form submission | **Yes** | User needs to know their submission failed |
| Unread notification count changes | **Yes** | Dynamic counter that updates without user action on it |
| "Items in cart" badge updates | **Yes** | Status indicator that changes based on user actions elsewhere |
| Page content loads initially | **No** | Initial content is read by the screen reader on focus |
| User navigates to a new screen | **No** | Screen readers announce the new screen title automatically |
| Progress bar updates | **Depends** | Announce milestones (25%, 50%, 100%) but not every pixel change |

<div className="cb-callout cb-callout--amber">
  <div className="cb-callout__header">Live Regions Are Not Free</div>
  <p>Every live region update interrupts whatever the screen reader is currently saying. Use them for important status changes that the user needs to know about, not for decorative or non-essential updates. Too many live region announcements create a noisy, confusing experience.</p>
</div>

## Part 1: Live Regions in Flutter

In Flutter, you create a live region by wrapping a widget with the `Semantics` widget and setting `liveRegion: true`:

```dart
Semantics(
  liveRegion: true,
  child: Text('Invalid email or password'),
),
```

When the `Text` inside this `Semantics` widget changes (or when the widget first appears), the screen reader announces the new text content.

### Testing Live Regions

To verify that a widget is a live region in a test, use `tester.getSemantics()` to get the semantics node and check the `SemanticsFlag.isLiveRegion` flag:

```dart
final semantics = tester.getSemantics(find.text('Invalid email or password'));
expect(
  semantics.getSemanticsData().hasFlag(SemanticsFlag.isLiveRegion),
  isTrue,
);
```

This reads the semantics tree (the accessibility tree that assistive technologies consume) and checks whether the node has the live region flag set.

<div className="cb-callout cb-callout--blue">
  <div className="cb-callout__header">Remember: ensureSemantics()</div>
  <p>Before any test that reads the semantics tree, call <code>handle = tester.ensureSemantics()</code> at the start and <code>handle.dispose()</code> at the end. This enables the semantics tree, which is normally disabled in tests for performance. Without it, <code>tester.getSemantics()</code> will throw.</p>
</div>

      handle.dispose();

      handle = tester.ensureSemantics();

      await tester.pumpWidget(
        const ProviderScope(

      // Screen readers will not announce it when it appears.

      expect(errorText, findsOneWidget);

## Key Concepts

| Concept | What It Does |
|---------|-------------|
| `Semantics(liveRegion: true)` | Marks a widget as a live region -- screen readers automatically announce content changes |
| `SemanticsFlag.isLiveRegion` | The flag to check in tests to verify a widget is a live region |
| `tester.getSemantics(finder)` | Returns the `SemanticsNode` for the matched widget -- requires `ensureSemantics()` |
| `tester.ensureSemantics()` | Enables the semantics tree in tests (normally disabled for performance) |
| WCAG 4.1.3 (Status Messages) | Status messages must be programmatically determinable without receiving focus |

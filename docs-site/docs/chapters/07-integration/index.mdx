# Chapter 7: Integration Tests — The Full Journey

> *"A screen reader user doesn't test one screen at a time. They navigate from login to dashboard to transactions. Your tests should do the same."*

**Estimated time:** ~30 minutes | **Focus:** Cross-screen tests | **Branch:** `chapter-7-integration`

---

## Why Integration Tests?

In Chapters 1 through 6, every test focused on a single screen. You pumped `LoginScreen` or `NotificationsScreen` in isolation and checked its accessibility properties. This is effective for catching bugs within a screen, but it misses an entire category of problems.

Real users navigate a journey:

1. They land on the Login screen and sign in
2. They arrive at the Dashboard and hear the greeting
3. They tap an account card and land on Transactions
4. They navigate via the bottom bar to Notifications

Accessibility issues can appear between screens -- focus might not move to the new screen, a heading might not exist on the landing page, or a navigation element might lose its label during a route transition. Integration tests catch these cross-screen issues that widget tests cannot see.

### Widget Tests vs. Integration Tests

| Aspect | Widget Test | Integration Test |
|--------|------------|-----------------|
| Scope | One widget or screen | Full app, multiple screens |
| Navigation | Not tested (screens pumped in isolation) | Full GoRouter navigation |
| Speed | Fast (~ms) | Slower (~seconds) |
| Runs on | Test runner only | Requires a device or emulator |
| Best for | Component-level accessibility | Journey-level accessibility |

## Setting Up the Integration Test

Flutter integration tests live in a top-level `integration_test/` directory (not inside `test/`). The `integration_test` package is already in the project's `pubspec.yaml` under `dev_dependencies`.

Create the file `integration_test/accessibility_audit_test.dart`:

<div className="cb-step">
  <div className="cb-step__badge">1</div>
  <div className="cb-step__content">
    <div className="cb-step__title">Initialize the integration binding</div>

Every integration test must start by initializing the integration test binding. This replaces the normal test binding with one that can control a real device:

```dart title="integration_test/accessibility_audit_test.dart"
import 'dart:ui';
import 'package:flutter/material.dart';
import 'package:flutter/semantics.dart';
import 'package:flutter_test/flutter_test.dart';
import 'package:integration_test/integration_test.dart';
import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'package:checkbank/app.dart';

void main() {
  IntegrationTestWidgetsFlutterBinding.ensureInitialized();

  testWidgets('full user journey with accessibility checks', (tester) async {
    final handle = tester.ensureSemantics();

    await tester.pumpWidget(const ProviderScope(child: CheckBankApp()));
    await tester.pumpAndSettle();

    // Tests go here...

    handle.dispose();
  });
}
```

Notice three things. First, **`IntegrationTestWidgetsFlutterBinding.ensureInitialized()`** replaces the default binding. This is required before any `testWidgets` call in an integration test. Second, **`ProviderScope(child: CheckBankApp())`** pumps the full app, not a single screen. The `CheckBankApp` includes `GoRouter`, the bottom navigation shell, and all routing logic. Third, **`tester.ensureSemantics()`** enables the semantics tree just like in widget tests, so we can audit accessibility properties.

</div>

## What This Journey Reveals

This single test exercises the entire main navigation path of the app. At each screen, you can add accessibility assertions:

- Does the screen have a heading?
- Do tappable elements have labels?
- Are live regions present where expected?
- Does focus move to the new screen after navigation?

The test itself is not about finding new bugs (the widget tests already do that). It is about verifying that the journey holds together -- that navigation transitions work, that screens render correctly when reached through the router, and that accessibility properties survive route changes.

## Key Concepts

| Concept | What It Does |
|---------|-------------|
| `IntegrationTestWidgetsFlutterBinding` | Replaces the default test binding to support running on a real device or emulator |
| `ProviderScope(child: CheckBankApp())` | Pumps the full app with all routing and state management intact |
| `tester.pumpAndSettle()` | Waits for all animations and async work to complete before assertions |
| `find.byTooltip('Back')` | Finds the AppBar back button by its tooltip text |
| Bottom nav `find.text('Alerts')` | Navigates between screens using the bottom navigation bar labels |
